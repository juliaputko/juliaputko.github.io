Add support for Multiple Databases in SmartSim

VIEW GITHUB DOCUMENTATION


While working on SmartSim, I implemented functionality to automate the deployment of multiple Redis databases, supporting workloads that require multiple Orchestrators for a Experiment.

Use case example: A workload may consist of a simulation with high inference performance demands (necessitating a co-located deployment), along with an analysis and visualization workflow connected to the simulation (requiring a standalone Orchestrator), which requires support for launching multiple databases.

The feature enables experiments to have multiple databases running. This involves users passing a database identifier to methods create_database, colocate_db_tcp and colocate_db_uds. The db_intentifier argument is required when initializing an Orchestrator or colocated Model during a multiple Orchestrator Experiment. SmartSim will set environment variables SSDB and SR_DB_TYPE with the identifier as a suffix. Communication with the Orchestrator via a SmartRedis Client is then made possible bu reading the SSDB and SR_DB_TYPE suffix.

Preview Entity Launch Details Prior to Start on Experiment

VIEW GITHUB DOCUMENTATION


While working on SmartSim. I led the design and core framwork enabling users to inspect the properties of hybrid HPC/AI workflow.

I added the ``Experiment.preview`` method to display the entity summaries during runtime to offer additional insight into the launch details, before actually launching any Experiment.

The method is crucial for the understanding Entities and their properties set by the suer A user can preview what they will be launching and make sure that all their assumptions on what has been set is correct prior to using and HPC resources.

The method surfaces entity information such as name, path, run settings, and client configuration of any instance of a ``Model``, ``Ensemble``, or ``Orchestrator`` prior to the start on an Experiment.

Any instance of a Model, Ensemble, or Orchestrator created by the Experiment can be passed as an argument to the preview method. Additionally, users may specify the name of a file to write preview data to via the output_filename argument, as well as the text format through the output_format argument. Users can also specify how verbose the preview is via the verbosity_level argument.

I built primarily in Python with Jinja templates, and the solution runs on major Linux HPC platforms, and integrates with workload managers including Slurm PBS, and LSF.

Framework for dataset conversion

GITHUB DOCUMENTATION


The ability to visualize and interpret simulation data in real time is valuable in understanding the behavior of what is going on in a simulation model, for example the behavior of a physical system. A SmartSim Dataset is a multidimensional array (tensor) in Fortran, C, C++, python with associated metadata) Currently SmartSim users can only retrieve SmartSim datasets and must unpack tensors and metadata using the dataset API to incorporate smartsim into their current workflows, The purpose of creating a data conversion method is to alleviate this requirement and lower technical barriers to integrating data visualization and analysis. Basically, the capabilities are there, it is just not as compact and usable as it could be. The dataformat being prioritized for these purposes is Xarray, a python package and data format commonly used in earth sciences, machine learning and visualization for large data. Xarray provides capabilities for data manipulations, analysis and visualizations, as well as convenient conversions into other formats for additional data analysis and storage (netcdf). An Xarray dataarray is basically a numpy- like array with dimensions, coordinates and attributes. Similar to pandas but with additional attached metadata One of the intended uses for what i was working on is to provide scientists a flexible, easy to use method for interacting with the data generated by numerical simulation models while they are running In my work I was taking a multidimensional array with its associated metadata posted from a simulation - and converting it into a format that is convenient for plotting in a jupyter notebook Or with other visualization tools Expanding the ability do do visualizations and analysis of data As an intern, derived technical requirements from data format standards to understand engineering restrictions for the conversion of HPE SmartRedis proprietary datasets to common data formats: VTK, netCDF, HDF5, Xarray. Implemented prescriptive methods in C, C++, FORTRAN, and Python to ensure that users added the necessary metadata to convert datasets streamed from C, C++, and FORTRAN numerical simulations into a broadly-used community format (i.e. Xarray), enabling online analysis and visualization of data.

Dataset conversion refers to a multi-step workflow where a user can create a SmartRedis dataset which can then be retrieved on the Python side, and then transformed into a {Dataformat} object.

I implemented the DatasetConverter class with add_metadata_for_xarray and transform_to_xarray method. Dataset conversion support has only been added for Xarray. Additional metadata is added to an existing SmartRedis dataset to add the possibility of conversion in the add_metadata_for_xarray method. The transform_to_xarray method ingests a Dataset and returns a dictionary of Xarray Dataarrays.

I performed extensive research on other data formats that should be added into the DatasetConverter class and options for how they would be added and the user types that would benefit from their addition

Integration of Humanoid Animation with Footsteps Navigation for Realistic Crowd Simulation

DOWNLOAD PDF


As an undergraduate I worked with the GAIDD lab #link on research for modelling human footsteps, and work towards effective crowd simulation.

Being able to model a realistic animation of a human on top of a footstep plan that adheres to biological constraints is a crucial step to being able to model crowds. This paper focuses on realism in agent representation as well as in agent navigation with the intention of producing realistic crowd models that can be used to represent real diverse crowds for a variety of applications. Decisions to combine a footstep model and realistic humanoid animation are explored by researching various methods related to character animation, crowd simulation and navigation control in a multi-agent dynamic environment. This paper records the work done to begin the integration of the virtual humans from the work in realistic humanoid animation including full body biped inverse kinematics, and barycentric blending with a footstep based navigation planner. Integrating navigation derived from multiple agent reinforcement learning is also explored. The result of the current work hopes to lead to humanoid animations that follow realistic navigation paths, are able to resolve collisions in a natural manner, and can represent diverse human motion. An investigation into various analysis techniques are proposed that can be later utilized to compare the completed current method against other methods to further confirm its usefulness.

This paper explores an animation technique, an inverse kinematic technique, and a navigation technique to begin an integration in order to make progress towards producing a crowd simulation of agents that can perform locomotion within the constraints of human motion. Justification behind each technique was provided in the context of choosing methods that were successful in producing natural human motion. Separate evaluations concluded that, separately, the techniques described produce movement that resembles natural looking movement, and can be implemented on scales needed for crowd simulation. Progress was made to attempt integration of these techniques with the assumption that a combination of the techniques within a single crowd simulation would also produce results that align with the goal of producing a crowd simulation that resembles natural movement. Evaluation techniques were explored in order to validate a working integration in the future.

Integrating sea-ice biogeochemistry into the public domain GOTM model framework

Arctic climate changes rapidly, therefore its modelling tools have to change at an even faster rate. In the jungle of updates, and the race to catch up with the latest submodules available for each part of the climate system, we often notice that the tradeoffs lead to a reduction of complexity.

Over the past few months, we worked on bringing up to date a set of relatively sophisticated sea ice and arctic biogeochemistry models, coupled to the well-known and widely used General Ocean Turbulence Model (GOTM). They contain state of the art models designed to simulate primary production (NPZD), as well as the carbon system and DMS fluxes in the Arctic. No new model has been implemented, so we will refer to the original publications and authors for further details and a review of their scientific validity. Namely: 1. 1D thermodynamic sea ice model, adapted from Flato & Brown (1996) by N. Steiner 2. Parametrizations for subgrid snow distribution and melt ponds (Abraham et al. 2015) 3. 1D ice algae model, adapted from Lavoie et al. (2005) by Steiner et al. 4. NPZD, DIC and DMS models for Resolute (CAA), by Hayashida & Mortenson (2017)
