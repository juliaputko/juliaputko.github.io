Add support for Multiple Databases in SmartSim

GITHUB DOCUMENTATION


While working on SmartSim, I implemented functionality to automate the deployment of multiple Redis databases, supporting workloads that require multiple Orchestrators for a Experiment.

Use case example: A workload may consist of a simulation with high inference performance demands (necessitating a co-located deployment), along with an analysis and visualization workflow connected to the simulation (requiring a standalone Orchestrator), which requires support for launching multiple databases.

The feature enables experiments to have multiple databases running. This involves users passing a database identifier to methods create_database, colocate_db_tcp and colocate_db_uds. The db_intentifier argument is required when initializing an Orchestrator or colocated Model during a multiple Orchestrator Experiment. SmartSim will set environment variables SSDB and SR_DB_TYPE with the identifier as a suffix. Communication with the Orchestrator via a SmartRedis Client is then made possible bu reading the SSDB and SR_DB_TYPE suffix.

Preview Entity Launch Details Prior to Start on Experiment

GITHUB DOCUMENTATION


While working on SmartSim. I led the design and core framwork enabling users to inspect the properties of hybrid HPC/AI workflow. I added the ``Experiment.preview`` method to display the entity summaries during runtime to offer additional insight into the launch details, before actually launching any Experiment.

The method is crucial for the understanding Entities and their properties set by the suer A user can preview what they will be launching and make sure that all their assumptions on what has been set is correct prior to using and HPC resources. The method surfaces entity information such as name, path, run settings, and client configuration of any instance of a ``Model``, ``Ensemble``, or ``Orchestrator`` prior to the start on an Experiment. Any instance of a Model, Ensemble, or Orchestrator created by the Experiment can be passed as an argument to the preview method. Additionally, users may specify the name of a file to write preview data to via the output_filename argument, as well as the text format through the output_format argument. Users can also specify how verbose the preview is via the verbosity_level argument.

I built primarily in Python with Jinja templates, and the solution runs on major Linux HPC platforms, and integrates with workload managers including Slurm PBS, and LSF.

Framework for Dataset Conversion

GITHUB DOCUMENTATION


The ability to visualize and interpret simulation data in real time is crucial for understanding the behavior of complex models—such as physical or climate simulations. A SmartSim Dataset is a multidimensional array (tensor) in Fortran, C, C++, or Python, accompanied by metadata.

Before the implementation, SmartSim users had to manually unpack these tensors and metadata through the dataset API to incorporate SmartSim into their workflows. To simplify this process, I developed a data conversion workflow that lowers technical barriers to visualization and analysis. The primary data format prioritized was Xarray—a Python library and data structure widely used in earth sciences, machine learning, and visualization for large datasets. Xarray provides intuitive tools for data manipulation, analysis, and conversion to formats such as netCDF for efficient storage and accessibility. I derived technical requirements from data format standards to understand engineering constraints for converting HPE SmartRedis proprietary datasets into community-standard formats (VTK, netCDF, HDF5, and Xarray). I implemented prescriptive methods in C, C++, Fortran, and Python to ensure consistent metadata integration and streamline interoperability between simulations and analysis tools.

I designed and implemented the DatasetConverter class, introducing the add_metadata_for_xarray and transform_to_xarray methods. These enable users to convert multidimensional datasets into Xarray DataArrays, allowing seamless plotting and real-time interaction within environments like Jupyter Notebooks. Additionally, I researched extensions for other scientific data formats and proposed implementation pathways for future integration, expanding the flexibility of SmartSim’s dataset ecosystem.

Integration of Humanoid Animation with Footsteps Navigation for Realistic Crowd Simulation

CPROOF

KADAPT GITHUB


As an undergraduate, I worked with the GAIDD Lab on research for modeling human footsteps and advancing effective crowd simulation. Being able to model realistic human animation on top of a biologically-constrained footstep plan is crucial for simulating believable crowds.

This paper focused on realism in agent representation and navigation, combining footstep modeling and humanoid animation techniques such as full-body biped inverse kinematics and barycentric blending, with navigation control in a multi-agent dynamic environment. Integration with reinforcement learning–based navigation was also explored.

The goal was to produce humanoid agents capable of natural locomotion, realistic collision avoidance, and diverse motion, enabling large-scale crowd simulations grounded in human biomechanics. Evaluation and analysis techniques were proposed to validate integration of these methods.

In preparation, I researched and presented prior work on reinforcement learning, gait analysis, and motion synthesis, and implemented initial tests in the Unity game engine.

GOTM

GOTM SEA ICE INTEGRATION GOTM FABM INTEGRATION GOTM CASES



I contributed to the General Ocean Turbulence Model (GOTM), a 1-D ocean model designed to simulate the physical behavior of a water column over time using weather and forcing data. GOTM reproduces underwater physical conditions such as temperature, salinity, and turbulence, providing a foundation for studying marine and climate processes.

My work focused on porting and integrating sea-ice and biogeochemistry models—originally used in Canada’s national climate model—into the open-source GOTM framework. This included the UVic Ice and UVic FABM (Framework for Aquatic Biogeochemical Models) modules, which were brought into compliance with GOTM’s latest structure and shared publicly via GitHub.

The integration supported simulations of Arctic climate impacts on marine and sea-ice ecosystems, coupling modules for NPZD, DIC, and DMS fluxes in Arctic regions. I collaborated with researchers to ensure reproducibility and open-source accessibility of the updated modules for the broader GOTM community.
